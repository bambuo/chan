## 概述

本文档定义缠论（Chan Theory）核心算法的量化实现规范，作为 TradeX 系统中信号引擎的算法基石。

缠论由缠中说禅（笔名）创立，是一套基于**价格形态**和**力度对比**的技术分析理论。其核心思想是：市场走势由趋势和盘整构成，通过识别分型、笔、线段、中枢等结构要素，可以客观地定位买卖点。

### 文档约定

为避免理论定义、工程近似和策略参数混用，本文档采用以下约定：

- **必须**：理论定义或实现不变量，违反即视为算法错误
- **应**：推荐实现方式，允许在有充分说明和测试覆盖时调整
- **可**：工程扩展或策略参数，不属于缠论原文的强制条件
- **理论规则**：来自缠论原文定义或可由原文直接推出的规则
- **工程规则**：为量化、实时计算、风险控制、测试便利而引入的实现约束，必须在输出或配置中标注为工程近似

### 数学符号约定

为保证实现可测试，本文档对可公式化的规则统一使用以下符号：

- 第 `i` 根经包含处理后的 K 线记为 `K_i = [L_i, H_i]`，其中 `H_i` 为最高价，`L_i` 为最低价，且 `H_i >= L_i`
- 任意走势段、线段或特征元素 `A` 的价格区间记为 `I(A) = [low(A), high(A)]`
- 两个闭区间重叠记为 `overlap(A, B) := max(low(A), low(B)) <= min(high(A), high(B))`
- 多个区间的公共重叠记为 `Overlap(S_1...S_n) = [max(low(S_i)), min(high(S_i))]`，当左端点 `<=` 右端点时重叠成立；其存在性记为 `has_overlap(S_1...S_n)`
- 方向函数：对最近一对非包含区间 `K_j -> K_i`，若 `H_i >= H_j`，记为向上；若 `L_i <= L_j`，记为向下；若边界相等导致方向仍不唯一，应继续向前寻找最近可判定的非包含关系
- 缠论级别记为 `L(n)`，交易所数据窗口记为 `interval`；二者不得直接等同

### 数字货币术语约定

本文档面向数字货币市场实现时，采用以下领域术语，避免把股票市场的交易制度和结算语义误植到加密资产交易系统中：

- 交易对象统一称为**交易对**、**交易标的**或**合约**；现货和杠杆交易使用交易对/交易标的，衍生品使用合约/合约标的，只有引用缠论原文标题时保留“股票”表述
- K 线周期在交易所接口中称为 **interval**，如 `1m`、`5m`、`30m`、`1h`、`1d`；缠论级别仍使用 `1F`、`5F`、`30F`、`日线级别` 等理论术语
- `Open` / `Close` 表示一个 interval 内的**周期起始价**和**周期结束价**，不隐含传统交易所的集合竞价、开盘、收盘制度
- 现货市场中，“买点”通常对应增持基础资产或由计价资产切换为基础资产；“卖点”通常对应减仓、空仓或切换为计价资产
- 合约市场中，应区分 U 本位/币本位、线性合约/反向合约；“做多/做空”分别对应开多、平空、开空、平多等方向性动作，是否使用杠杆、逐仓/全仓保证金模式和资金费率过滤由风控与执行模块决定
- 成交量需区分 base volume（基础资产成交量）、quote volume（计价资产成交额）、turnover（交易所定义的成交额/成交名义价值）和 trade count（成交笔数）；不同交易所、不同合约类型的 `volume` / `turnover` 单位可能不同，数据适配层必须保留原始单位并输出归一化口径

### 范围与非目标

**范围：**
- 单级别处理链路：K 线包含处理 → 分型 → 笔 → 线段 → 中枢 → 走势类型 → 背驰 → 买卖点
- 多级别联立的工程近似方案
- 中间结构的输出要求，便于测试、回放和可视化调试
- 数字货币现货与合约市场中的信号语义映射

**非目标：**
- 不承诺用固定交易所 interval 严格替代缠论级别
- 不将 MACD、成交量、评分权重等工程过滤项视为缠论原文必要条件
- 不在本文档中定义具体目标仓位、止损、订单执行、杠杆倍数、保证金模式和风控执行策略

## 一、Kline 数据模型

### 1.1 原始 Kline

```go
type Kline struct {
    Time        time.Time // 时间戳
    Open        float64   // 周期起始价
    High        float64   // 最高价
    Low         float64   // 最低价
    Close       float64   // 周期结束价
    BaseVolume  float64   // 基础资产成交量；若交易所原始字段非 base 口径，应由数据适配层归一化
    QuoteVolume float64   // 计价资产成交额；若交易所不提供则置 0 并在元数据中标注
    Turnover    float64   // 成交额/成交名义价值；单位以交易所 product/category/instrumentType 定义为准
    TradeCount  int64     // 成交笔数；交易所未提供时为 0
    RawVolumeUnit   string // 原始 volume 字段单位，如 "base"、"quote"、"contract"、"coin"
    RawTurnoverUnit string // 原始 turnover 字段单位，如 "base"、"quote"、"contract"、"coin"
}
```

> **数据适配要求**：`Kline` 是算法层统一使用的 K 线数据结构；`BaseVolume`、`QuoteVolume`、`Turnover` 是算法层使用的归一化字段。原始交易所字段必须在适配层保留来源与单位。线性合约与反向合约的 `volume` / `turnover` 单位可能相反，算法不得仅凭字段名推断成交量含义。

### 1.2 K 线包含处理

包含关系定义：相邻两根 K 线，一根的高低点完全在另一根的高低点范围之内。

**包含关系公式：**

对相邻两根 K 线 `K_a = [L_a, H_a]`、`K_b = [L_b, H_b]`：

```
contains(K_a, K_b) := (H_a >= H_b && L_a <= L_b) || (H_b >= H_a && L_b <= L_a)
```

**包含处理规则（向上处理）：**
- 前提：前一非包含 K 线对构成向上关系（按原文口径，后一根 K 线高点不低于前一根高点）
- 规则：取两根 K 线高点的最大值、低点的最大值，合并为一根新 K 线

```
merge_up(K_a, K_b) = [max(L_a, L_b), max(H_a, H_b)]
```

**包含处理规则（向下处理）：**
- 前提：前一非包含 K 线对构成向下关系（按原文口径，后一根 K 线低点不高于前一根低点）
- 规则：取两根 K 线高点最小值、低点最小值，合并为一根新 K 线

```
merge_down(K_a, K_b) = [min(L_a, L_b), min(H_a, H_b)]
```

**处理顺序：**
1. 从左到右逐根处理
2. 先判断方向（向上还是向下），再执行包含处理
3. 方向由最近一个**非包含关系**的 K 线对确定
4. 处理后产生的新 K 线继续与后续 K 线比较，直到不存在包含关系
5. **边界情况**：若序列起始处无法确定方向（如前两两包含），保留第一根 K 线不变，不做方向性合并

```
包含处理示例（向上处理）：
K1: H=100, L=90
K2: H=95,  L=92   ← K2 被 K1 包含
合并: H=100, L=92
```

## 二、分型

### 2.1 顶分型

连续三根 K 线（经包含处理后），中间 K 线的最高点是三根中最高的，最低点也是三根中最高的。

```
     H2(最高)
    / \
   /   \
H1     H3
L1     L3
    L2(最高)
```

### 2.2 底分型

连续三根 K 线（经包含处理后），中间 K 线的最低点是三根中最低的，最高点也是三根中最低的。

```
    L2(最低)
    / \
   /   \
L1     L3
H1     H3
    H2(最低)
```

### 2.3 分型公式

对连续三根经包含处理后的 K 线 `K_{i-1}, K_i, K_{i+1}`：

```
top_fractal(i) :=
    H_i > H_{i-1} && H_i > H_{i+1}
 && L_i > L_{i-1} && L_i > L_{i+1}

bottom_fractal(i) :=
    L_i < L_{i-1} && L_i < L_{i+1}
 && H_i < H_{i-1} && H_i < H_{i+1}
```

若实现需要处理相等价格，应在配置中定义严格比较或非严格比较；理论默认采用严格几何关系，避免同一组三根 K 线同时产生歧义分型。

### 2.4 分型结构

```go
type FractalType int

const (
    FractalNone    FractalType = 0
    TopFractal     FractalType = 1  // 顶分型
    BottomFractal  FractalType = 2  // 底分型
)

type Fractal struct {
    Type     FractalType  // 分型类型
    Index    int          // 在原始 K 线序列中的位置
    High     float64      // 分型最高价（顶分型的顶）
    Low      float64      // 分型最低价（底分型的底）
    Strength float64      // 分型强度（K 线实体比例）
}
```

### 2.5 分型确认规则

**重要区分：分型的存在 vs 分型用于笔**

- **分型的存在**：分型由 K 线形态客观决定，一旦连续三根经包含处理后的 K 线满足顶/底分型条件，分型即**客观存在**，无需等待任何确认
- **分型用于笔**：在构筑笔时，采用交替出现规则——向上笔连接底分型→顶分型，向下笔连接顶分型→底分型，方向必须严格交替

**分型区间（Fractal Range）：**
- 顶分型区间：[max(L1, L3), H2] — 由第一、三根K线低点中的较高者到中间K线高点
- 底分型区间：[L2, min(H1, H3)] — 由中间K线低点到第一、三根K线高点中的较低者

```
top_range(i)    = [max(L_{i-1}, L_{i+1}), H_i]
bottom_range(i) = [L_i, min(H_{i-1}, H_{i+1})]
```

**用于成笔的分型筛选规则：**
1. 用于构成一笔的顶底分型之间至少间隔 1 根独立 K 线（既不属于前一分型也不属于后一分型的 K 线）
2. 用于构成一笔的两个分型不能共用 K 线
3. 分型的第三根 K 线如果出现跳空缺口，可作为分型强度更高的辅助特征
4. 同一方向上，如果连续出现多个同类型分型，取最极端的那个（最高顶或最低底）作为笔的端点候选

## 三、笔

### 3.1 定义

笔是顶分型与底分型之间的连线，代表一个**最小级别的趋势方向**。

```
底分型
  |  \
  |   \   ← 向上笔：底分型 → 顶分型
  |    \
  |    顶分型

顶分型
  |\
  | \   ← 向下笔：顶分型 → 底分型
  |  \
  |  底分型
```

### 3.2 笔的成立条件

1. 相邻的顶分型和底分型之间至少包含 **1 根独立 K 线**（不含分型本身的 K 线）
2. 分型之间不能共用 K 线
3. 笔的方向交替出现（向上 → 向下 → 向上 → ...）

**成笔公式：**

设分型 `F_a`、`F_b` 分别位于 K 线索引 `idx(F_a)`、`idx(F_b)`，分型类型 `type(F) in {Top, Bottom}`：

```
opposite(F_a, F_b) := type(F_a) != type(F_b)
independent_k_count(F_a, F_b) := idx(F_b.left) - idx(F_a.right) - 1
valid_bi(F_a, F_b) := adjacent_extreme(F_a, F_b)
                     && opposite(F_a, F_b)
                     && independent_k_count(F_a, F_b) >= 1

direction(F_a, F_b) = +1, if type(F_a)=Bottom && type(F_b)=Top
direction(F_a, F_b) = -1, if type(F_a)=Top && type(F_b)=Bottom
```

其中 `adjacent_extreme` 表示经过同向分型取极值和无效分型忽略后，二者是相邻的有效顶/底分型。

> **工程扩展说明**：以上为本文档默认成笔标准（分型间至少 1 根独立 K 线），与原文基础定义保持一致。若策略需要更强过滤，可通过 §12 配置 `BiMinKLineCount` 等参数启用”严笔/新笔”工程扩展；该扩展属于实现侧过滤规则，输出和测试中必须与理论默认标准区分。

### 3.3 笔的破坏

- 向上笔被**向下笔**破坏（出现顶分型 → 底分型构成的向下笔）
- 向下笔被**向上笔**破坏（出现底分型 → 顶分型构成的向上笔）

### 3.4 特殊规则：笔的延伸

- 若出现新的分型但未构成反向笔，当前笔继续延伸
- 笔的端点取最高/最低点，而非分型的周期结束价

```go
type Bi struct {
    StartIndex int       // 笔起始 K 线索引
    EndIndex   int       // 笔结束 K 线索引
    Direction  int       // 1=向上, -1=向下
    StartPrice float64   // 起始价格
    EndPrice   float64   // 结束价格
    High       float64   // 笔内最高价
    Low        float64   // 笔内最低价
    Length     float64   // 笔的幅度 (|EndPrice - StartPrice|)
    Slope      float64   // 笔的斜率 (Length / K线数量)
    KLineCount int       // 笔包含的 K 线数量
}
```

### 3.5 构线段前的特征序列包含处理

在构筑线段之前，需要对用于线段判定的**特征序列元素**进行包含处理。这里的“笔的包含处理”并不是指常规笔序列中允许连续同方向笔；标准笔序列仍应由顶底分型交替构成、方向交替出现。包含处理主要发生在线段特征序列中：向上线段取其向下笔序列，向下线段取其向上笔序列，再按包含关系合并这些特征元素。

**特征元素包含的定义：**
- 特征元素 A 包含特征元素 B：`high(A) >= high(B) && low(A) <= low(B)`；或 B 以同样方式包含 A
- 若两个特征元素高低点完全相同，视为边界相等的退化包含，工程实现应保留来源索引并按稳定规则合并，不得由此制造新的方向

**处理规则（按非包含特征元素对的方向，与K线包含处理逻辑一致）：**
- **向上关系（当前元素高点不低于前一元素高点）**：取高高（max(H1,H2)）、高低（max(L1,L2)），合并为一个特征元素
- **向下关系（当前元素低点不高于前一元素低点）**：取低高（min(H1,H2)）、低低（min(L1,L2)），合并为一个特征元素

**处理要点：**
1. 从左到右依次处理相邻特征元素的包含关系
2. 方向由**最近非包含特征元素对**的高点/低点关系确定（而非线段方向或笔自身方向）
3. 包含处理后的特征序列用于后续的特征序列分型判定

> 工程实现可以保留特征元素的来源笔索引，便于回溯线段端点；但理论层面应以“特征序列元素”的包含处理为准，而不是把常规笔序列改造成连续同向笔序列。

## 四、线段

### 4.1 定义

线段由至少三笔构成，是工程递归中构造次级别走势类型的基础结构。线段的成立前提是起始前三笔存在重叠部分；线段的确认和破坏不应简单等同于“三笔重叠区域”。“前三笔重叠”是线段成立前提，“至少三个连续次级别走势类型重叠”才是中枢定义，二者不得混用。

```
       笔3
      /   \
     /     \
笔1 /       \
   / \     笔4
  /   \    /
 /     \  /
/ 笔2   \/
         笔5

线段起始至少包含三笔，且前三笔必须有重叠部分；
后续通过特征序列判断延续或破坏
```

### 4.2 线段成立前提

线段至少由三笔构成，但任意连续三笔并不必然构成线段；只有起始前三笔存在共同重叠部分，才构成线段起点。

```
valid_segment_start(B_1, B_2, B_3) :=
    has_overlap(B_1, B_2, B_3)
 := max(low(B_1), low(B_2), low(B_3))
 <= min(high(B_1), high(B_2), high(B_3))
```

该重叠只用于确认线段起始结构，不生成中枢对象；中枢必须由至少三个连续次级别走势类型的重叠部分构成。

### 4.3 线段的起点

- 从任意一笔的起点开始
- 该笔的方向决定线段的初始方向

### 4.4 特征序列

**特征序列是线段划分的核心工具**，用于判定线段何时被破坏。

**定义：**
- **向上线段的特征序列**：线段内的**向下笔**序列（第2、4、6...笔）
- **向下线段的特征序列**：线段内的**向上笔**序列（第2、4、6...笔）

**特征序列的包含处理：**
1. 提取特征序列中的笔元素
2. 对相邻特征序列元素按方向进行包含处理（规则同K线包含处理——方向由特征序列元素间的**非包含关系**确定，而非线段本身的方向）
   - 当特征序列元素间构成向上关系时：取高高、高低
   - 当特征序列元素间构成向下关系时：取低高、低低
3. 包含处理后的特征序列元素等同于K线，可以寻找顶底分型

**特征序列分型：**
- 向上线段的特征序列（向下笔序列）出现标准**顶分型** → 向上线段被确认破坏
- 向下线段的特征序列（向上笔序列）出现标准**底分型** → 向下线段被确认破坏
- 分型的第三元素完成后，线段结束

**特征序列公式：**

设线段由笔序列 `B_1...B_n` 构成，`direction(B_i) in {+1, -1}`：

```
segment_up(B_1...B_n)   := direction(B_1)=+1
feature_seq_up          := [B_i | i>=2 && direction(B_i)=-1]

segment_down(B_1...B_n) := direction(B_1)=-1
feature_seq_down        := [B_i | i>=2 && direction(B_i)=+1]

gap(E_i, E_{i+1}) := !overlap(I(E_i), I(E_{i+1}))
```

对标准特征序列 `E_1...E_m`，其分型判定复用 K 线分型公式：

```
feature_top(i) :=
    high(E_i) > high(E_{i-1}) && high(E_i) > high(E_{i+1})
 && low(E_i)  > low(E_{i-1})  && low(E_i)  > low(E_{i+1})

feature_bottom(i) :=
    low(E_i)  < low(E_{i-1})  && low(E_i)  < low(E_{i+1})
 && high(E_i) < high(E_{i-1}) && high(E_i) < high(E_{i+1})
```

```go
// 特征序列元素：线段中用于判定破坏的笔
type FeatureElement struct {
    Bi       Bi      // 对应的笔
    StartIdx int     // 起始K线索引
    EndIdx   int     // 结束K线索引
    High     float64 // 元素高点
    Low      float64 // 元素低点
}
```

### 4.5 线段的延续

- 线段未被破坏前持续延伸
- 向上笔开始的线段，可表示为底顶序列 d1-g1-d2-g2-...-dn-gn；若存在 j >= i+2 且 dj <= gi，则向上线段发生笔破坏
- 向下笔开始的线段，可表示为顶底序列 g1-d1-g2-d2-...-gn-dn；若存在 j >= i+2 且 gj >= di，则向下线段发生笔破坏
- 线段破坏的精确定义以后续特征序列分型为准；笔破坏条件用于描述线段可能结束的几何前提

### 4.6 线段的破坏

线段被破坏有**两种情况**（缠论原文第67课精确定义）：

**情况一：特征序列无缺口的标准破坏**

特征序列经包含处理后出现标准分型，且该分型的**第一元素与第二元素之间不存在价格缺口**（有重叠）：
- 向上线段：特征序列（向下笔序列）出现顶分型，第一、二元素间无缺口 → 线段在该顶分型的高点处结束
- 向下线段：特征序列（向上笔序列）出现底分型，第一、二元素间无缺口 → 线段在该底分型的低点处结束

> **定理（第67课）：** 在线段破坏的第一种情况下，必然是笔破坏的。即：情况一的破坏必然伴随笔破坏。

**情况二：特征序列有缺口的破坏（需二次确认）**

特征序列出现分型，但该分型的**第一元素与第二元素之间存在价格缺口**（无重叠）：
1. 不能立即判定线段结束，需等待后续走势的确认
2. 从该分型的极点开始，观察**新形成的特征序列**
3. 若新的特征序列出现反向分型（不分第一种还是第二种情况，只要有分型即可），**原线段在第一次分型的极点处结束**

> **定理（第67课）：** 在第二种情况下，后一特征序列不一定封闭前一特征序列相应的缺口。且第二个序列中的分型，不分第一二种情况，只要有分型就可以。

> **定理（第71课）：** 线段破坏如果不是笔破坏，那么一定是第二种情况。反之亦然——情况二的破坏不一定伴随笔破坏。

```
情况一（无缺口，标准破坏）：
向上线段 [笔1↑, 笔2↓, 笔3↑, 笔4↓, 笔5↑, 笔6↓]
特征序列 = [笔2↓, 笔4↓, 笔6↓]
           ↓ 包含处理后出现顶分型，且顶分型第一、二元素间无缺口
           → 线段在该顶分型高点处立即结束

情况二（有缺口，需二次确认）：
向上线段 [笔1↑, 笔2↓, 笔3↑, 笔4↓, 笔5↑, 笔6↓, 笔7↑, 笔8↓]
特征序列 = [笔2↓, 笔4↓, 笔6↓, 笔8↓]
           ↓ 包含处理后出现顶分型，但第一、二元素间有缺口
           ↓ 从顶分型高点开始的向下笔线下跌，新特征序列中出现底分型
           → 原线段在第一次顶分型的高点处结束
```

## 五、中枢

### 5.1 定义

中枢是**至少三个连续次级别走势类型的重叠部分**，代表多空力量均衡区。工程实现中，若当前算法层级尚未完整递归出次级别走势类型，可使用已确认线段作为次级别走势类型的承载结构，但输出必须标注该口径属于工程归一化。

```
次级别走势类型 Z_1: 从 A 到 B
次级别走势类型 Z_2: 从 B 到 C
次级别走势类型 Z_3: 从 C 到 D

通用三段重叠校验 = [max(各次级别走势类型低点), min(各次级别走势类型高点)]
               = [max(d1, d2, d3), min(g1, g2, g3)]

按原文 Z 走势段口径简化计算：
  ZG（中枢上沿）= min(g1, g2)  ← 按原文 Z 走势段口径形成后锁定
  ZD（中枢下沿）= max(d1, d2)  ← 按原文 Z 走势段口径形成后锁定
  GG（波动最高）= max(g_n)     ← 随延伸更新
  DD（波动最低）= min(d_n)     ← 随延伸更新
```

**原文记法与工程口径：**
- 原文以与中枢形成方向一致的次级别走势类型作为 Z 走势段，记其高低点为 `gn/dn`
- 原文定义：`ZG = min(g1, g2)`，`ZD = max(d1, d2)`，中枢区间为 `[ZD, ZG]`
- 工程实现可在已归一化的三段线段上用 `[max(L1,L2,L3), min(H1,H2,H3)]` 校验重叠区间，但必须保证与 Z 走势段定义得到的 `[ZD, ZG]` 一致

**中枢公式：**

设中枢由连续次级别走势类型 `Z_1...Z_n` 构成，`g_i = high(Z_i)`，`d_i = low(Z_i)`：

```
ZG = min(g_1, g_2)
ZD = max(d_1, d_2)
pivot_range = [ZD, ZG]

GG_n = max(g_1, g_2, ..., g_n)
DD_n = min(d_1, d_2, ..., d_n)
fluctuation_range_n = [DD_n, GG_n]

valid_pivot(Z_1, Z_2, Z_3) :=
    consecutive(Z_1, Z_2, Z_3)
 && same_sub_level(Z_1, Z_2, Z_3)
 && overlap(I(Z_1), I(Z_2))
 && overlap(I(Z_2), I(Z_3))
 && max(d_1, d_2, d_3) <= min(g_1, g_2, g_3)
 && ZD <= ZG
```

若工程实现以三段线段 `S_1,S_2,S_3` 作为承载结构，则必须满足：

```
engineering_overlap(S_1,S_2,S_3) :=
    max(low(S_1), low(S_2), low(S_3)) <= min(high(S_1), high(S_2), high(S_3))
```

### 5.2 中枢的级别

中枢的级别由构成它的次级别走势类型决定：
- `L(n)` 中枢 = 至少三个连续 `L(n-1)` 走势类型的重叠部分
- 最低不可再分级别的中枢，可按至少三个该级别单位 K 线重叠部分定义
- 工程实现中若使用线段承载次级别走势类型，应在结果中保留 `sourceLevel` 与 `isEngineeringProxy` 标记
- 更高级别中枢的形成，应以同级别中枢波动区间重叠或递归构筑结果为准，不应简单写成“三个本级别中枢重叠”

```
pivot_level(P) = L(n), if P = overlap(T_1, T_2, T_3, ...), T_i in L(n-1)
```

### 5.3 中枢的延伸、扩展与扩张

中枢形成后有三种演变方式，对应不同的市场状态。**关键原则：ZG/ZD 形成后永不改变，仅 GG/DD 随延伸更新。**

**中枢延伸：**
- 后续次级别走势 `Z_n` 与中枢区间 `[ZD, ZG]` 仍有重叠，未形成第三类买卖点
- ZG/ZD **保持不变**，仅 GG/DD 随新的 Z 走势段高/低点扩展
- 中枢级别保持不变
- 对应市场围绕中枢继续震荡蓄力；不要求每一段走势完全落在 [ZD, ZG] 内

```
extend(P, Z_n) := overlap(I(Z_n), [ZD, ZG])
               := d_n <= ZG && g_n >= ZD
```

**中枢扩展：**
- 若采用后续通行口径，可将同级别中枢长期延伸形成更高级别中枢视为工程判定规则
- 该规则必须标注为工程规则或后续口径，不应作为第20课原文精确定理直接引用
- 是否以“9 段”作为升级阈值，应配置化，并由测试用例单独覆盖

**中枢扩张：**
- 价格突破原中枢区间，形成第二个中枢
- 第二个中枢的波动区间与第一个中枢的波动区间**发生重叠**
- 两个本级别中枢叠加产生更高级别中枢
- 对应走势从本级别升级到更高级别

```
enlarge(P_a, P_b) :=
    level(P_a) = level(P_b)
 && overlap([DD_a, GG_a], [DD_b, GG_b])
 && !overlap([ZD_a, ZG_a], [ZD_b, ZG_b])  // 若中枢区间直接重叠，则不构成趋势中的两个独立同级别中枢
```

```
中枢延伸：                     中枢扩张：
  ┌─────┐                      ┌─────┐
  │ 中枢 │ ←仍与[ZD,ZG]重叠       │中枢1│   ┌──中枢2──┐
  └─────┘                      └─────┘  /           \
   级别不变                           \  /  重叠区域  \
                                    ────── 级别+1 ──────
```

### 5.4 中枢的破坏（第三类买卖点）

**第三类买卖点定理（缠论第20课精确定义）：**

> 一个次级别走势类型向上离开缠中说禅走势中枢，然后以一个次级别走势类型回试，其低点不跌破 ZG，则构成第三类买点；一个次级别走势类型向下离开缠中说禅走势中枢，然后以一个次级别走势类型回抽，其高点不升破 ZD，则构成第三类卖点。

**精确定义（两段结构）：**

三买 = **第一段（离开段）**向上离开中枢 + **第二段（回抽段）**向下回试不跌破 ZG
三卖 = **第一段（离开段）**向下离开中枢 + **第二段（回抽段）**向上回抽不升破 ZD

```
third_buy(P, Leave, Test) :=
    direction(Leave) = up
 && leave_up(P, Leave)
 && direction(Test) = down
 && low(Test) >= ZG
 && first_test_after_leave(P, Leave, Test)

third_sell(P, Leave, Test) :=
    direction(Leave) = down
 && leave_down(P, Leave)
 && direction(Test) = up
 && high(Test) <= ZD
 && first_test_after_leave(P, Leave, Test)

leave_up(P, Leave)   := end_price(Leave) > ZG && final_part_no_overlap_above(P, Leave)
leave_down(P, Leave) := end_price(Leave) < ZD && final_part_no_overlap_below(P, Leave)
```

其中 `final_part_no_overlap_above/below` 是工程判定函数，用于表达离开段末端已经脱离中枢区间：

```
final_part_no_overlap_above(P, Leave) :=
    exists suffix S of Leave:
        low(S) > ZG && !overlap(I(S), [ZD, ZG])

final_part_no_overlap_below(P, Leave) :=
    exists suffix S of Leave:
        high(S) < ZD && !overlap(I(S), [ZD, ZG])
```

严格递归实现中，`Leave` 本身应是一个次级别走势类型；若工程上用线段承载，应在结果中标注工程代理口径。

**关键要点：**
- 第三类买卖点确认位置在**回抽段确认不跌破/不升破中枢边界之后**，而非离开段刚离开时
- 必须是**第一次**回抽——并非任何回调/反弹都是第三类买卖点
- 回抽段必须是**次级别走势类型**；工程实现中可由已确认线段或低级别走势类型承载，但不得把未确认的单根 K 线波动直接当成回抽段
- 第三类买卖点确认中枢被破坏（PivotDestroyed）

### 5.5 中枢生命周期

```
状态机：形成中 → 已形成 → 延伸中 → 工程扩展升级(可选阈值) / 扩张升级(同级别中枢波动区间重叠) / 被破坏(三买/三卖)
```

```go
// 中枢状态枚举
type PivotState int

const (
    PivotForming   PivotState = 0 // 形成中（连续次级别走势类型不足3个）
    PivotFormed    PivotState = 1 // 已形成（至少3个连续次级别走势类型重叠）
    PivotExtending PivotState = 2 // 延伸中
    PivotExpanded  PivotState = 3 // 工程扩展升级（如启用延伸段数阈值，级别+1）
    PivotEnlarged  PivotState = 4 // 扩张升级（两中枢波动区间重叠，级别+1）
    PivotDestroyed PivotState = 5 // 被第三类买卖点破坏
)

type Pivot struct {
    StartIndex   int       // 中枢起始K线索引
    EndIndex     int       // 中枢结束K线索引
    ZG           float64   // 中枢上沿 = min(g1,g2) —— 按 Z 走势段形成后锁定不变
    ZD           float64   // 中枢下沿 = max(d1,d2) —— 按 Z 走势段形成后锁定不变
    GG           float64   // 波动最高点 = max(g_n) —— 随延伸/扩展更新
    DD           float64   // 波动最低点 = min(d_n) —— 随延伸/扩展更新
    Segments     []Segment // 工程承载结构：构成中枢的线段或低级别走势类型
    OverlapCount int       // 重叠走势段数量
    Level        int       // 中枢级别（1=本级别，2=高级别...）
    State        PivotState // 中枢当前状态
    Direction    Direction  // 中枢方向：上涨中枢(Z段=向下段) / 下跌中枢(Z段=向上段)
}
```

**关键区分：中枢区间 vs 波动区间**

| 概念 | 区间 | 定义 | 是否变化 |
|------|------|------|---------|
| **中枢区间** | [ZD, ZG] | ZG = min(g1,g2), ZD = max(d1,d2) | **形成后永不改变** |
| **波动区间** | [DD, GG] | GG = max(g_n), DD = min(d_n) | 随延伸/扩展更新 |

- **ZG/ZD** 用于判断第三类买卖点（回抽是否进入中枢）
- **GG/DD** 用于判断中枢扩张（两个中枢波动区间是否重叠，产生更高级别中枢）

## 六、走势类型与走势必完美

### 6.1 走势必完美定理

**走势必完美**是缠论的第一原理，是所有买卖点存在的基础：

> 任何级别的任何走势类型终将完成。
> - 上涨趋势必以**顶背驰**结束
> - 下跌趋势必以**底背驰**结束
> - 盘整走势完成时通常通过**第三类买卖点**转出，或通过延伸、扩展、扩张升级为更高级别结构

该定理回答了「任何走势结构都要走向完成」这一根本问题，是整个量化体系的基石。量化实现中应避免把“盘整完成”机械等同为“必然立即出现第三类买卖点”，还需处理中枢延伸、扩展、扩张等级别升级路径。

### 6.2 走势类型分类

走势类型是策略框架的顶层设计——确定当前走势类型后，才能确定适用的交易策略。

**上涨趋势：**
- 包含两个或以上依次向上的中枢
- 后一同级别中枢的 `DD` **高于**前一同级别中枢的 `GG`，即两个中枢的波动区间没有重叠
- 对应策略：现货可偏向持有或增持基础资产，合约可偏向开多或减空，直到出现顶背驰或风控条件触发

**下跌趋势：**
- 包含两个或以上依次向下的中枢
- 后一同级别中枢的 `GG` **低于**前一同级别中枢的 `DD`，即两个中枢的波动区间没有重叠
- 对应策略：现货可偏向减仓、空仓或持有计价资产，合约可偏向开空或减多，直到出现底背驰或风控条件触发

**盘整：**
- 仅包含一个中枢
- 价格在中枢区间内波动
- 对应策略：在中枢边界附近做调仓决策；现货表现为基础资产与计价资产的比例调整，合约表现为多空方向或净敞口调整，直到出现第三类买卖点或中枢延伸、扩展、扩张导致级别变化

**走势类型公式：**

设完成走势类型 `T` 包含同级别中枢序列 `P_1...P_m`：

```
range_type(T) := m = 1

up_trend(T) :=
    m >= 2
 && forall i in [1, m-1]:
        level(P_i) = level(P_{i+1})
     && DD_{i+1} > GG_i

down_trend(T) :=
    m >= 2
 && forall i in [1, m-1]:
        level(P_i) = level(P_{i+1})
     && GG_{i+1} < DD_i
```

其中 `DD/GG` 使用中枢波动区间；若两个同级别中枢波动区间重叠，应按中枢扩张处理，而不是判为趋势延续。

```
上涨趋势：
  中枢1[DD1, GG1]   中枢2[DD2, GG2]
  ┌──────────┐     ┌──────────┐
  │          │ ↗  │          │ ↗
──┘          └──┘          └──→  DD2 > GG1

下跌趋势：
  中枢1[DD1, GG1]   中枢2[DD2, GG2]
  ┌──────────┐     ┌──────────┐
  │          │ ↘  │          │ ↘
──┘          └──┘          └──→  GG2 < DD1

盘整：
  中枢[ZD, ZG]
  ┌──────────┐
  │  震荡区域  │
──┘          └──
```

### 6.3 走势类型数据结构

```go
type TrendType int

const (
    TrendUp    TrendType = 1  // 上涨趋势（2个或以上同向中枢波动区间无重叠）
    TrendDown  TrendType = -1 // 下跌趋势（2个或以上同向中枢波动区间无重叠）
    RangeOnly  TrendType = 0  // 盘整（仅1个中枢）
)

type Trend struct {
    Type           TrendType  // 走势类型
    Pivots         []Pivot    // 构成走势的中枢列表
    StartIndex     int        // 走势起始K线索引
    EndIndex       int        // 走势结束K线索引
    IsComplete     bool       // 走势是否已完成（走势必完美）
    CompleteReason string     // 完成原因：顶背驰 / 底背驰 / 三买 / 三卖
}
```

## 七、背驰

### 7.1 定义

背驰是**价格与力度之间的背离**，是判断走势是否衰竭的核心依据。

### 7.2 背驰的级别

背驰的可靠性由级别决定，不可混用：

| 级别 | 判定对象 | 可靠性 | 应用场景 |
|------|---------|--------|---------|
| 笔背驰 | 相邻同向笔之间的力度对比 | 低（信号频繁） | 短线辅助参考 |
| 线段背驰 | 线段内部两段之间或相邻线段的力度对比 | 中 | 波段操作 |
| 走势背驰 | 趋势中最后一个中枢前后的力度对比 | 高（信号稀缺） | 中周期开仓/平仓或现货仓位切换 |

```go
type DeviationLevel int

const (
    BiDeviation      DeviationLevel = 0 // 笔背驰（信号最弱）
    SegmentDeviation DeviationLevel = 1 // 线段背驰（信号中等）
    TrendDeviation   DeviationLevel = 2 // 走势背驰（信号最强）
)
```

### 7.3 背驰的计算

**力度指标：**
- 缠论理论上要求比较走势力度的强弱，但不限定唯一数值指标
- 工程实现必须将力度度量抽象为 `force_metric`，可选实现包括价格斜率、MACD 红绿柱面积、MACD 黄白线高度等
- `price_force` 是默认工程度量之一，不是缠论原文的唯一标准

```
price_force(A) = abs(end_price(A) - start_price(A)) / max(1, kline_count(A))

macd_area(A) = sum(abs(hist_t)), t in A
macd_peak(A) = max(abs(DIF_t)), t in A

weaken(A_before, A_after, metric) := metric(A_after) < metric(A_before)
force_metric in {price_force, macd_area, macd_peak, custom_metric}
```

**盘整背驰：**
```
在一个中枢的结构上下文中，比较中枢前后两段走势的力度
前段力度: A1 = ΔPrice / ΔTime
后段力度: A2 = ΔPrice / ΔTime
若 A2 < A1 且价格创新高/低 → 盘整背驰
```

```
range_top_deviation(P, A_before, A_after) :=
    high(A_after) > high(A_before)
 && weaken(A_before, A_after, force_metric)

range_bottom_deviation(P, A_before, A_after) :=
    low(A_after) < low(A_before)
 && weaken(A_before, A_after, force_metric)
```

**趋势背驰：**
```
在至少包含两个同向中枢的趋势结构中，比较进入段和离开段
第一个中枢前的趋势力度: M1
第二个中枢后的趋势力度: M2
若 M2 < M1 且价格创新高/低 → 趋势背驰
```

```
trend_top_deviation(P_1, P_2, A_enter, A_leave) :=
    level(P_1) = level(P_2)
 && trend_direction(P_1, P_2) = up
 && high(A_leave) > high(A_enter)
 && weaken(A_enter, A_leave, force_metric)

trend_bottom_deviation(P_1, P_2, A_enter, A_leave) :=
    level(P_1) = level(P_2)
 && trend_direction(P_1, P_2) = down
 && low(A_leave) < low(A_enter)
 && weaken(A_enter, A_leave, force_metric)
```

**结构前提：**
- 背驰不是孤立的相邻线段数值比较，必须依附于完整走势结构
- 趋势背驰至少需要两个同向、同级别中枢作为结构基础
- 盘整背驰必须围绕一个已形成中枢比较其前后走势力度

```go
type Deviation struct {
    Type           string         // "trend" 趋势背驰 | "range" 盘整背驰
    Level          DeviationLevel // 背驰级别：笔背驰 / 线段背驰 / 走势背驰
    Direction      int            // 1=顶背驰, -1=底背驰
    SegmentBefore  *Segment       // 被比较的前一段走势
    SegmentAfter   *Segment       // 被比较的后一段走势
    PriceHigh      float64        // 价格高点
    ForceBefore    float64        // 前段力度值
    ForceAfter     float64        // 后段力度值
    MACDAreaBefore float64        // 前段 MACD 面积
    MACDAreaAfter  float64        // 后段 MACD 面积
    MACDDiffBefore float64        // 前段 MACD DIF 极值
    MACDDiffAfter  float64        // 后段 MACD DIF 极值
}
```

### 7.4 MACD 辅助判断规则

MACD 是背驰判断的最常用辅助工具。以下规则属于量化工程中的确认条件，可按策略风格配置，不应理解为缠论原文的唯一必要条件：

**MACD 面积计算：**
- 前段 MACD 面积 = 前段走势对应区间内 MACD 红绿柱的绝对值之和
- 后段 MACD 面积 = 后段走势对应区间内 MACD 红绿柱的绝对值之和
- 面积缩小 = 力度衰减的信号

**MACD 黄白线（DIF/DEA）辅助判定：**
- 比较前段和后段 DIF 的极值（绝对值）
- 后段 DIF 极值的绝对值 < 前段 DIF 极值的绝对值 → 力度衰减
- **常用过滤条件**：DIF 有效回抽 0 轴（或接近 0 轴）之后再发散，可提升背驰确认质量
  - 量化参考：回抽到位时 |DIF| 应低于近期 DIF 波动幅度的 10%~15%（如 |DIF| < ATR(DIF, N) × 0.15），或 DIF 与 DEA 交叉（DIF×DEA≤0）视为有效回抽。具体阈值可在 §12 设计约束中配置

**MACD 背离工程确认条件（可配置）：**
1. MACD 面积缩小（后段面积 < 前段面积）
2. MACD 黄白线高度降低（后段 DIF 极值 < 前段 DIF 极值）
3. 价格创新高/低（后段绝对价格位置 > 前段，但力度反之）

### 7.5 区间套定位

**区间套是缠论精确定位买卖点转折位置的核心技术**。

**定义：**
从高级别确认背驰后，逐级下钻到更低级别，在更低级别上用同样的背驰判定找到精确的转折点。

**定位流程：**
```
日线背驰确认 → 30F 级别背驰确认 → 5F 级别背驰确认 → 1F 级别精确点定位

每一级结构判定需满足：
1. 走势结构完整（中枢完成 + 离开段出现）
2. 本级按同一结构规则确认背驰成立
3. 本级确认后背驰成立，继续下钻到次级别重复判定
4. 最低级别背驰点即为精确的买卖点

可选工程过滤：
- MACD 辅助确认（面积缩小 + DIF 背离 + 回抽0轴）
- 成交额、盘口深度、滑点、资金费率等数字货币市场因子
```

**量化实现要点：**
- 区间套的核心是**递归背驰判定**——每一级别的判定逻辑完全相同，只是数据源不同
- 最低级别通常取到 1F 笔或 1F 线段的背驰点
- 若某一级别未能确认背驰，则区间套链条断裂，低级别精确定位失败或降级；这会降低操作精度，但不必然否定高级别背驰本身

```go
type IntervalNestingResult struct {
    Levels        []string    // 参与的级别列表，如 ["1d", "30m", "5m", "1m"] 或 ["日线", "30F", "5F", "1F"]
    Confirmations []Deviation // 每一级确认的背驰
    FinalIndex    int         // 最终精确定位的K线索引
    FinalPrice    float64     // 最终精确定位的价格
    Accuracy      float64     // 定位精度评分 (0~1)
}
```

## 八、买卖点

### 8.1 买卖点与级别的对应关系

买卖点必须标注级别才有操作意义，不同级别的买卖点对应完全不同的交易机会。下表只描述工程观察窗口和风险预算倾向，不直接生成订单：

| 买卖点观察窗口（工程近似） | 交易节奏 | 持有/暴露时长 | 风险预算倾向 |
|-----------|---------|---------|---------|
| `1d` / 日线窗口一买一卖 | 中周期方向切换 | 以天到周计 | 较高风险预算 |
| `30m` / 30F 窗口一买一卖 | 波段调仓 | 以小时到天计 | 中等风险预算 |
| `5m` / 5F 窗口一买一卖 | 短线调仓 | 以分钟到小时计 | 较低风险预算 |
| `1m` / 1F 窗口买卖点 | 超短线/日内/高频候选 | 以分钟计 | 极低风险预算 |

**级别对应关系：**
- 严格理论中，买卖点级别由递归构筑关系决定，而非固定交易所 interval 决定
- 高级别的买卖点是低级别走势类型的完成点
- 工程实现可近似使用“`1d` / 日线窗口 ↔ `30m` / 30F 窗口 ↔ `5m` / 5F 窗口 ↔ `1m` / 1F 窗口”下钻定位，但必须在输出中标注这是观察窗口近似
- 订单方向、目标仓位比例、杠杆倍数、保证金模式、止损止盈和滑点控制属于执行与风控模块，不属于买卖点算法本身

### 8.2 第一类买卖点

**第一类买点（一买）：**
- 下跌走势类型完成并转化的关节点
- 典型表现为下跌趋势末端出现底背驰
- 最安全的买点，但需要等待确认

```
buy_point_1(T, x) :=
    trend_direction(T) = down
 && trend_bottom_deviation(T)
 && trend_complete(T)
 && x = endpoint(T)
```

**第一类卖点（一卖）：**
- 上涨走势类型完成并转化的关节点
- 典型表现为上涨趋势末端出现顶背驰

```
sell_point_1(T, x) :=
    trend_direction(T) = up
 && trend_top_deviation(T)
 && trend_complete(T)
 && x = endpoint(T)
```

其中 `trend_complete(T)` 表示走势类型已经满足“走势终完美”的结构完成条件；工程实现可按以下规则判定：

```
trend_complete(T) :=
    complete_structure(T)
 && (
        trend_top_deviation(T)
     || trend_bottom_deviation(T)
     || transformed_to_other_type(T)
    )
```

`complete_structure(T)` 至少要求该走势类型包含一个同级别中枢；趋势走势还应满足至少两个同向同级别中枢的结构条件。`transformed_to_other_type` 用于表达走势完成后转化为盘整、上涨或下跌中的另一类走势，不应依赖订单执行结果。

### 8.3 第二类买卖点

**第二类买点（二买）：**
- 第一类买点之后，价格二次回踩不破前低
- 次级别出现底分型或底背驰
- 比一买确定性更高

```
buy_point_2(B1, Pullback, x) :=
    after(Pullback, B1)
 && low(Pullback) >= price(B1)
 && lower_level_confirm_bottom(Pullback)
 && x = endpoint(Pullback)
```

**第二类卖点（二卖）：**
- 第一类卖点之后，价格二次反弹不破前高
- 次级别出现顶分型或顶背驰

```
sell_point_2(S1, Rebound, x) :=
    after(Rebound, S1)
 && high(Rebound) <= price(S1)
 && lower_level_confirm_top(Rebound)
 && x = endpoint(Rebound)
```

其中 `lower_level_confirm_bottom/top` 表示次级别回调或反弹走势已经完成，并出现相应方向的转折确认；确认方式可来自次级别走势类型完成、分型、背驰或其他可配置工程信号，不限定为单一形态。

### 8.4 第三类买卖点

**第三类买点（三买）：**
- 次级别走势向上离开中枢后，次级别回试低点不跌破 ZG
- 回试完成处可结合底分型、次级别背驰等信号确认

```
buy_point_3(P, Leave, Test, x) :=
    third_buy(P, Leave, Test)
 && x = endpoint(Test)
```

**第三类卖点（三卖）：**
- 次级别走势向下离开中枢后，次级别回抽高点不升破 ZD
- 回抽完成处可结合顶分型、次级别背驰等信号确认

```
sell_point_3(P, Leave, Test, x) :=
    third_sell(P, Leave, Test)
 && x = endpoint(Test)
```

**第三类买卖点工程确认：**
- `lower_level_confirm_bottom(Test)` 可作为三买的工程增强确认，但不属于三买理论定义的必要条件
- `lower_level_confirm_top(Test)` 可作为三卖的工程增强确认，但不属于三卖理论定义的必要条件
- 底分型、顶分型、次级别背驰、成交额或盘口确认只能影响信号强度或执行过滤，不应改变三买/三卖是否成立的理论判定

```go
type SignalType int

const (
    BuyPoint1  SignalType = 1  // 一买
    BuyPoint2  SignalType = 2  // 二买
    BuyPoint3  SignalType = 3  // 三买
    SellPoint1 SignalType = -1 // 一卖
    SellPoint2 SignalType = -2 // 二卖
    SellPoint3 SignalType = -3 // 三卖
)

type Signal struct {
    Type      SignalType       // 信号类型
    Level     string           // 信号级别（如 "30F", "5F"）
    Index     int              // 信号出现位置
    Price     float64          // 信号价格
    Strength  float64          // 信号强度 (0~1)
    Pivot     *Pivot           // 关联的中枢
    Deviation *Deviation       // 关联的背驰
    Nesting   *IntervalNesting // 关联的区间套定位
}
```

### 8.5 买卖点之间的转化关系

三类买卖点并非孤立存在，它们之间存在重要的转化关系：

**二三买合并（最强买入信号）：**
- 一买之后，次级别回抽既没有跌破一买低点（满足二买条件），又恰好在中枢上沿之上回抽完成（满足三买条件）
- 二买和三买重合在同一位置 → 信号强度极高
- 对应走势：中枢破坏后的强势确认

**二买本质：**
- 高级别的二买 = 次级别的一买或二买
- 例如：日线级别的二买，在30F级别上是一个完整的一买走势

**三买转一卖（风险转化）：**
- 三买之后，若离开中枢的走势力度不济（出现盘整背驰），三买可能转化为一卖
- 判断依据：三买后的离开段是否形成新的中枢并出现顶背驰
- 量化中需要实时监测三买后的力度变化

**买卖点共振（多级别信号叠加）：**
- 同一时刻不同级别同时出现同向买卖点 → 信号共振，可靠性倍增
- 例如：`1d` 三买 + `30m` 一买 + `5m` 一买同时触发 → 多头共振信号；现货可解释为增持/持有基础资产，合约可解释为开多/减空候选

## 十、多级别联立

缠论的一个重要思想是**多级别联立分析**。

### 10.1 级别的本质

**级别 ≠ 交易所 interval。** 级别由走势的递归构筑定义，不随观察窗口改变。

> `1m` K 线可以走出 5F 甚至 30F 级别的走势（当波动足够大时）。级别是走势自身的属性，而非 K 线 interval 的属性。

但在实际量化中，为了工程可行性，通常使用交易所 interval 作为级别的近似观察窗口。以下是常见的近似映射：

| 交易所 interval（近似窗口） | 缠论级别 | 递归构筑关系 |
|-------------------|---------|-------------|
| `1m` | 1F 级别 | 1F K线 → 1F 笔 → 1F 线段 → 1F 中枢 → 1F 走势类型 |
| `5m` | 5F 级别 | 5F 笔 = 1F 线段，5F 线段 = 1F 走势类型 |
| `30m` | 30F 级别 | 30F 笔 = 5F 线段，30F 线段 = 5F 走势类型 |
| `1d` | 日线级别 | 日线笔 = 30F 线段，日线线段 = 30F 走势类型 |

> **重要提示**：上表仅为工程实现的近似参考。实际级别应由递归构筑结果决定，**不以 interval 为准**。在极端波动行情下，`1m` K 线可能直接构成 5F 甚至 30F 级别的走势，此时应以递归结果校正级别判断，而非机械依赖 interval 窗口。

### 10.2 完整级别递归定义

缠论的级别是通过递归从最低级别逐级构筑的：

```
级别 N 的构筑流程（从最低级别开始）：

Level 1 (1F):
  1F K线 → 1F 包含处理 → 1F 分型 → 1F 笔 → 1F 线段 → 1F 中枢 → 1F 走势类型

Level 2 (5F):
  5F 笔 = 1F 线段
  5F 线段 = 1F 走势类型
  5F 中枢 = 至少三个连续 1F 走势类型重叠（工程上可由已确认 5F 线段承载）
  5F 走势类型 = 5F 中枢与其前后同级别走势组合

Level 3 (30F):
  30F 笔 = 5F 线段（即 1F 走势类型）
  30F 线段 = 5F 走势类型
  ...

Level N+1:
  L(N+1)笔 = L(N)线段
  L(N+1)线段 = L(N)走势类型
```

**递归关键规则：**
- 高级别的一笔 = 低级别的一个线段
- 高级别的一个线段 = 低级别的一个走势类型
- 高级别的中枢 = 至少三个连续低级别走势类型重叠；若以线段作为承载结构，必须标注工程代理口径

### 10.3 多级别特征

训练时可以将多个级别的特征拼接：
```
[`1d`/日线特征 | `30m`/30F特征 | `5m`/5F特征 | 目标标签]
```

## 十一、信号强度评分

信号强度评分属于工程策略层，不属于缠论原文定义。默认实现可综合考虑以下因素，输出 0~1 的置信度；实际权重必须可配置，并在回测报告中记录配置版本。

| 因子 | 权重 | 说明 |
|------|------|------|
| 级别大小 | 0.3 | 级别越大，信号越可靠 |
| 背驰力度 | 0.25 | 力度差异越大，信号越可靠 |
| 多级别共振 | 0.2 | 多个级别同时出信号加分 |
| 流动性配合 | 0.15 | 成交额、盘口深度和预估滑点支持信号时加分 |
| 中枢位置 | 0.1 | 中枢边界附近的信号更可靠 |

数字货币市场可额外接入以下工程因子，但这些因子不改变缠论结构定义：

- 资金费率、基差、合约溢价率，用于过滤过度拥挤的合约方向信号
- 盘口深度、预估滑点、最小下单量、数量步进（lot size）、最小价格变动单位（tick size）、最小名义金额（min notional），用于判断信号是否具备可执行性
- quote volume、turnover、trade count、taker buy volume、异常放量，用于补充单纯 base volume 的流动性不足
- 杠杆倍数、保证金模式、强平距离、维持保证金率，用于执行层风险预算约束

### 11.1 ML 特征导出

每个买卖点信号（`Signal`）结构体包含 `Features map[string]float64` 字段，对齐 Python 参考实现的 `CFeatures`。内置 12 个特征：

| 特征名 | 信号类型 | 含义 |
|--------|---------|------|
| `bsp_bi_amp` | 通用 | 信号笔振幅 |
| `divergence_rate` | T1, T1P | MACD 背驰率 |
| `zs_cnt` | T1 | 中枢数量 |
| `bsp1_bi_amp` | T1P | 最后一笔振幅 |
| `bsp2_retrace_rate` | T2 | 回撤比率 |
| `bsp2_break_bi_amp` | T2 | 突破笔振幅 |
| `bsp2_bi_amp` | T2 | 二买笔振幅 |
| `bsp2s_retrace_rate` | T2S | 类二买回撤率 |
| `bsp2s_break_bi_amp` | T2S | 类二买突破振幅 |
| `bsp2s_bi_amp` | T2S | 类二买笔振幅 |
| `bsp2s_lv` | T2S | 级别 |
| `bsp3_zs_height` | T3 | 中枢相对高度 |
| `bsp3_bi_amp` | T3 | 三买笔振幅 |

可通过 `features.ExportToLibSVM()` 导出为 LIBSVM 格式，直接喂给 XGBoost/LightGBM 等框架。

## 十二、设计约束

### 12.1 输入约束

- 输入 K 线必须按时间升序排列
- 时间戳应使用交易所返回的标准时间，推荐在数据适配层统一为 UTC 毫秒或 `time.Time` 的 UTC 表示
- `High >= Low`，价格字段不得为 `NaN` 或无穷值
- 成交量、成交额、成交笔数不得为负数
- 数字货币为 24/7 连续交易市场，算法不得依赖股票市场的开收盘、午休、集合竞价或涨跌停制度
- 跳空可来自真实价格跃迁、交易所停服、数据缺失或流动性断层；实现应保留缺口信息，并在回放日志中区分数据缺口与价格缺口
- 若数据源同时提供 base volume、quote volume、turnover、trade count 或 taker buy volume，应在数据适配层分别保留；仅有单一 `volume` 字段时必须说明其计量单位与归一化方式
- 对 U 本位/线性合约、币本位/反向合约、交割合约和现货交易对，成交量与成交额字段的单位不得复用同一默认假设，必须按交易所产品类型解析
- 少于 3 根 K 线时，算法必须返回明确错误或空结构，不得产生伪分型

### 12.2 输出约束

算法结果必须输出以下中间结构，便于回放、测试和可视化：

- 包含处理后的 K 线序列
- 客观分型列表，以及用于成笔的分型筛选结果（若实现中区分）
- 笔序列
- 标准特征序列及其分型
- 线段、中枢、走势类型
- 背驰和买卖点信号

### 12.3 确定性

- 同一输入序列、同一配置必须产生完全相同的输出
- 算法不得依赖随机数、当前系统时间或非确定性遍历顺序
- 增量计算结果应与同一数据的全量计算结果保持一致；若因窗口截断只能保证近端一致，必须在接口文档中说明一致性边界

### 12.4 配置边界

| 配置项 | 类型 | 理论/工程属性 | 说明 |
|--------|------|---------------|------|
| `BiMinKLineCount` | int | 理论默认 + 工程扩展 | 默认按原文基础成笔要求；大于默认值时属于严笔过滤 |
| `EnableBiInclusion` | bool | 工程实现 | 若开启，其作用对象是特征序列元素，而非破坏常规笔序列交替性 |
| `MACDFastPeriod` / `MACDSlowPeriod` / `MACDSignalPeriod` | int | 工程过滤 | MACD 辅助背驰判断参数 |
| `BspMacdAlgo` / `BspMacdAlgoBuy` / `BspMacdAlgoSell` | string | 工程过滤 | 力度度量方法；可分别指定买/卖侧的算法（area/peak/slope/amp/…） |
| `BspMaxBs2sLv` | int | 工程过滤 | 类二买卖点最大搜索层级（距离 T2 的笔间隔数/2），nil 不限 |
| `EnableMultiLevel` | bool | 工程近似 | 多级别联立分析开关，不等同于严格递归级别实现 |
| `EnableSegBSP` | bool | 工程扩展 | 线段级买卖点检测开关，开启后将线段视为高级别笔独立检测信号 |
| `CalBoll` / `CalRsi` / `CalKdj` | bool | 工程过滤 | 分别控制布林带、RSI、KDJ 的计算（底层使用 talib 库，与 TA-Lib 标准对齐） |
| `MeanMetrics` / `TrendMetrics` | int[] | 工程过滤 | 均线周期列表 / 趋势力度指标周期列表 |
| `UpdateWindowSize` | int | 工程实现 | 增量更新尾部窗口大小（K线根数），默认 300 |
| `DebugFractalCheck` | bool | 调试 | 开启分型冲突检测日志 |
| `TriggerStep` | bool | 工程实现 | 启用逐 K 线增量分析模式 |

### 12.5 性能约束

- 全量计算应按输入长度线性或近线性增长
- 增量更新可使用固定窗口优化，但必须保留足够上下文以避免破坏分型、笔、线段和中枢的连续性
- 若为性能牺牲严格递归级别或全量一致性，必须在结果中暴露近似标记

### 12.6 测试要求

实现必须至少覆盖以下测试类型：

- K 线包含处理：向上、向下、连续包含、起始方向未知
- 分型识别：顶分型、底分型、无分型、同向分型筛选
- 成笔：方向交替、独立 K 线数、共用 K 线排除
- 线段：前三笔重叠、特征序列包含、第一种破坏、第二种破坏
- 中枢：ZG/ZD 锁定、GG/DD 更新、延伸、扩展、扩张、第三类买卖点
- 走势类型：盘整、上涨趋势、下跌趋势、走势完成状态
- 背驰与信号：趋势背驰、盘整背驰、MACD 工程过滤、区间套降级
- 增量一致性：增量结果与全量结果在声明窗口内一致
